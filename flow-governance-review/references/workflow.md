## Purpose

Use this skill to produce reusable flow-governance artifacts from local flow/process snapshots. The outputs are designed to be consumed by other skills or by OpenClaw, not just read as prose.

This skill intentionally keeps a narrow boundary:

- include duplicate evidence and canonical rewrite planning
- include alias-map generation after cleanup or version changes
- include process-flow reference scan, deterministic repair, and validation
- include bounded OpenClaw review-pack export/apply steps
- include optional publish of reviewed local rows through the explicit publish command
- exclude remote scope export and arbitrary remote CRUD outside the explicit publish command

## Runtime Model

- The official shell entrypoint is `scripts/run-flow-governance-review.sh <command> ...`.
- The shell wrapper resolves Python in this order: `FLOW_GOVERNANCE_PYTHON_BIN`, `PAB_PYTHON_BIN`, `process-automated-builder/.venv/bin/python`, then `python3`.
- Main commands are local-first. Live fetches or remote writes are opt-in and depend on env described in `references/env.md`.
- Some direct helper scripts import `tiangong_lca_spec` or `tidas_sdk`; run them with the `process-automated-builder` venv when in doubt.
- Not every script in this skill is a shell subcommand. `get-flow`, `list-flows`, `review-flows`, `remediate-flows`, `publish-version`, and `regen-product` are CLI-backed wrappers; auxiliary naming-completion and later remediation workflows remain direct Python entrypoints under `flow-governance-review/scripts/`.
- Do not invoke deleted compatibility wrappers under `process-automated-builder/scripts/`. The supported helper locations are the canonical scripts under `flow-governance-review/scripts/`.

## Artifact Layout

Persistent flow-processing outputs belong under `flow-governance-review/assets/artifacts/flow-processing/`.

Use these subtrees consistently:

- `datasets/` for reusable pools and canonical flow JSONL scopes
- `validation/` for grouped validation results that drive later remediation
- `naming/` for incomplete-name analysis, zero-process recovery, and OpenClaw completion packs
- `remediation/` for deterministic remediation and MCP sync outputs generated by the remediation helpers

Related retained bundles:

- `assets/artifacts/flow-processing/naming/zero-process-completion-pack/` for batched missing-name completion review/apply/publish artifacts
- `assets/artifacts/flow-remediation-batch-smoketest/` for historical smoke-test evidence around remediation-helper startup checks

Do not reintroduce these machine artifacts under `docs/`.

### `openclaw-entry`

Unified OpenClaw-facing wrapper for this skill.

The wrapper chooses the source mode from the arguments:

1. `--flow-id`: fetch the subject flow first, then run the UUID workflow
2. `--subject-flows-file` or `--flows-file`: run governance directly from the provided local snapshot
3. when a local subject snapshot is provided without `--candidate-flows-file` and no live candidate-pool env is available, fall back to subject-only dedup by reusing the subject snapshot as the candidate pool

Optional process-pool behavior:

- if `--process-pool-file` is provided, the wrapper reuses exact `process_id@version` rows from that local pool when they still reference the subject flow
- in UUID mode, the wrapper still performs the remote metadata scan to discover current process keys, but fetches full JSON only for keys missing from the local pool
- later deterministic repairs and OpenClaw ref-apply outputs can sync updated rows back into the same pool

Default output root when `--out-dir` is omitted:

- `~/.openclaw/workspace/review/flow-governance/<flow-id>` for UUID mode
- `~/.openclaw/workspace/review/flow-governance/local-<source-stem>-<timestamp>` for local snapshot mode

Primary outputs:

- `openclaw-handoff-summary.json`
- `openclaw-entry-command-result.json`

### `openclaw-full-run`

OpenClaw-friendly wrapper for the standard workflow when the caller starts with a flow UUID instead of local snapshots.

The wrapper:

1. fetches the subject flow for `--flow-id` through MCP `Database_CRUD_Tool` when available, with PostgREST as fallback
2. discovers visible processes that reference that flow UUID when authenticated PostgREST access is available
3. if `--process-pool-file` is provided, reuses matching exact-version rows from the local pool and fetches full JSON only for missing referencing processes
4. materializes those snapshots under `inputs/`
5. calls `run-governance` with the fetched snapshots
6. syncs deterministic repaired process rows back into the local process pool when applicable
7. writes `live-fetch-manifest.json` and `openclaw-handoff-summary.json`

Primary outputs:

- `inputs/live-subject-flows.snapshot.jsonl`
- `inputs/live-processes.snapshot.jsonl` when process fetch is enabled and rows were found
- `live-fetch-manifest.json`
- `openclaw-command-result.json`
- `openclaw-handoff-summary.json`

Recommended usage:

```bash
scripts/run-flow-governance-review.sh openclaw-full-run \
  --flow-id 4b5eadd2-816c-420c-b382-5e0a975b53a6
```

### `run-governance`

Preferred fixed orchestration entrypoint for this skill.

Use this when you want one standard local governance run instead of composing subcommands yourself. The orchestrator runs the applicable phases in this order:

1. `review-flows`
2. `flow-dedup-candidates`
3. `export-openclaw-dedup-review-pack` when semantic duplicate pairs remain
4. `build-flow-alias-map` when old/new scopes are provided
5. `scan-process-flow-refs` when `--processes-file` is provided
6. `plan-process-flow-repairs`
7. `apply-process-flow-repairs`
8. `validate-processes`
9. `export-openclaw-ref-review-pack` when residual ref decisions remain
10. `export-openclaw-text-review-pack` for the target flow scope
11. `export-openclaw-classification-review-pack` for flows with classification or `typeOfDataSet` findings

Important behavior:

- unless `--enable-review-llm` is passed, the orchestrator drives `review-flows` in rule-only mode
- if `--process-pool-file` is provided, the orchestrator passes it through to deterministic process-repair apply so `repair-apply/repair-summary.json` records `process_pool_sync`
- `governance-run-manifest.json` is the stable machine-readable run ledger and includes the actual subprocess commands, per-step status, and key outputs

Primary outputs:

- `governance-run-manifest.json`
- staged subdirectories under the run root: `inputs/`, `review/`, `dedup/`, `dedup-pack/`, `alias/`, `scan/`, `repair/`, `repair-apply/`, `validate/`, `ref-pack/`, `flow-text-pack/`, `flow-classification-pack/`

### `flow-dedup-candidates`

Generate duplicate-family evidence, canonical maps, rewrite plans, and manual-review queues for product/waste flows.

Important behavior:

- same-UUID version lineage is outside semantic dedup scope and does not produce auto-merge rewrite outputs
- cross-UUID high-similarity pairs are routed to semantic review instead of direct auto-merge
- if `--candidate-flows-file` is omitted in subject mode, the command fetches the candidate pool live and persists the exact snapshot it used
- live visible-process counts are optional and must be requested explicitly with `--live-ref-counts` when `--processes-file` is omitted

- `flow-dedup-summary.json`

### `plan-process-flow-repairs`

Generate a deterministic repair plan from process rows, scope flows, alias map, and optional scan findings.

Prefer `regen-product` when you want one canonical CLI-backed entrypoint for scan plus repair planning. Keep `plan-process-flow-repairs` only for lower-level helper use.

The auto-patch boundary is explicit:

- `--auto-patch-policy alias-only` is the default and recommended mode
- `--auto-patch-policy alias-or-unique-name` allows the exact-name fallback
- `--auto-patch-policy disabled` turns every non-trivial case into manual review

Primary outputs:

- `repair-plan.json`
- `repair-plan.jsonl`
- `manual-review-queue.jsonl`
- `repair-summary.json`

### `apply-process-flow-repairs`

Prefer `regen-product --apply` when you want one canonical CLI-backed entrypoint for scan, repair apply, and local validation. Keep `apply-process-flow-repairs` only for lower-level helper use.

When `--process-pool-file` is provided, the patched process rows are also merged back into that pool and `repair-summary.json` records `process_pool_sync`.

Primary outputs:

- `repair-plan.json`
- `repair-plan.jsonl`

### `publish-reviewed-data`

Prepare and optionally publish reviewed flow/process rows after OpenClaw decisions have been applied locally.

Important behavior:

- flow rows can publish with `skip`, `append_only_bump`, or `upsert_current_version`
- process rows can publish with `skip`, `append_only_bump`, or `upsert_current_version`
- if `--original-flow-rows-file` is provided, unchanged reviewed flow rows are skipped before planning publish, so they do not get version-bumped or dry-run publish reports
- when prepared published flow versions differ from the input versions, process `referenceToFlowDataSet` pointers can be rewritten to the published target versions before process publish
- dry-run is the default; remote writes happen only with `--commit`

Primary outputs:

- `prepared-flow-rows.json`
- `prepared-process-rows.json`
- `flow-version-map.json`
- `skipped-unchanged-flow-rows.json`
- `process-flow-ref-rewrite-evidence.jsonl`
- `publish-report.json`

## Recommended Sequences

### Standard local governance

1. Prefer `run-governance` when you already have local flow snapshots.
2. Inspect `review/flow_review_summary.json`, `dedup/flow-dedup-summary.json`, and `dedup/flow-snapshot-manifest.json`.
3. If semantic duplicate pairs remain, send only `dedup-pack/review-pack.json` to OpenClaw.
4. If process rows are in scope, inspect `scan/scan-summary.json`, `repair/repair-summary.json`, and `validate/validation-report.json`.
5. Export only the residual manual-review queues to OpenClaw.
6. Apply OpenClaw decisions locally and validate before publish.

### OpenClaw from a flow UUID

1. Prefer `openclaw-entry --flow-id` as the single OpenClaw-facing entrypoint.
2. Reuse one shared `--process-pool-file` across repeated UUID runs when you want to avoid refetching exact-version processes.
3. Use `openclaw-handoff-summary.json` as the stable machine-readable handoff artifact.

### Incomplete-name and zero-process recovery

1. Run `analyze_incomplete_flow_names_with_process_pool.py` to identify incomplete three-field names and sync process evidence.
2. If zero-process residuals remain, run `discover_zero_process_refs_via_mcp.py` to recover any still-visible references through MCP search plus strict verification.
3. Export the remaining zero-process naming scope with `export_missing_name_field_completion_pack.py`.
4. After OpenClaw writes per-batch decisions, run `run_missing_name_completion_postprocess.py` to apply, validate, aggregate, and optionally publish the accepted patches.

### Invalid-flow remediation batch

1. Run `scripts/run-flow-governance-review.sh remediate-flows` or `node scripts/run-remediate-flows.mjs` on the invalid flow pool.
2. Publish the ready subset with `scripts/run-flow-governance-review.sh publish-version` or `node scripts/run-publish-version.mjs`.
3. When the accepted flow changes require process-side reference regeneration, run `scripts/run-flow-governance-review.sh regen-product --apply` or `node scripts/run-flow-regen-product.mjs --apply`.
4. If remote validation failures remain, run `remediate_remote_validation_failed_flows_round2.py`.
5. Re-run `publish-version` only on the round-2 ready subset.
6. Use the residual manual queue prompts only for the rows that still fail after the deterministic passes.

### Publish after review

1. Use `publish-reviewed-data` only after local apply and validation steps are clean.
2. Pass `--original-flow-rows-file` when unchanged flows should be skipped entirely instead of being version-bumped.
3. Keep the default dry-run first, then repeat with `--commit` only for the reviewed rows that are actually ready to ship.

## Guardrails

- Keep local JSON payloads as the system of record for this skill.
- Prefer explicit snapshots over hidden live reads; the dedup command persists the exact snapshots it used.
- Use machine-readable artifacts as handoff boundaries; avoid burying decisions only in markdown.
- Treat `flow-alias-map.json` and explicit OpenClaw decision files as authoritative inputs for downstream patch steps.
- Treat `review-pack-context.json` as the authoritative context sidecar for flow naming, text, or classification review.
- Treat a shared `--process-pool-file` as a reusable local cache plus working set for exact-version process rows, not as a semantic merge layer across versions.
- Only use the explicit publish command for reviewed rows that are ready to ship.
- Do not use this skill for arbitrary remote CRUD outside the reviewed-row publish boundary.
- Do not route callers back to deleted `process-automated-builder/scripts/` wrapper paths; use the canonical scripts in this skill directly.
